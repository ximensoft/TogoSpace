# Multi-Agent 聊天系统 - 实施路线图

## 项目概述

构建一个支持多个 AI Agent 互相聊天的演示系统，每个 Agent 通过配置文件定义其角色和性格，支持在多个聊天室中并发通信。

> 说明：本文档中的 roadmap 内容默认以**后端服务能力**为范围，描述的是后端在调度、推理、持久化、可观测性与对外接口上的演进目标；前端（TUI / Web）相关工作仅在需要配合后端能力落地时做辅助性说明。

---

## 版本概览

### V1: 双 Agent 单房间

实现最简单的多 Agent 聊天场景，验证核心概念：

- 固定两个 Agent（Alice 和 Bob），每个通过配置文件定义
- 在单个聊天室中轮流对话
- 使用 asyncio 实现基础调度机制
- 固定轮次后自动结束对话
- 验证 Agent 能表现出配置的性格特征

### V2: 多 Agent 单房间

扩展 V1，验证架构的可扩展性：

- 支持从配置文件动态加载任意数量的 Agent（3+ 个）
- 实现基于 asyncio 的消息队列机制
- 在单个聊天室内，多个 Agent 轮流对话
- 验证调度器能正确处理多个 Agent 的并发请求

### V3: 多 Agent 多房间

扩展 V2，支持更复杂的场景：

- 支持创建多个独立的聊天室
- Agent 可以选择加入不同的聊天室
- Agent 可以同时在多个聊天室中活跃
- 消息正确路由到对应的聊天室
- Agent 能感知聊天室内的其他参与者
- 各聊天室之间数据完全隔离：同名 Agent 在不同房间是独立实例，各自只感知本房间的消息历史

### V3.1: 跨房间上下文感知

扩展 V3，打破房间数据隔离：

- 同时参与多个房间的 Agent 能感知其他房间的消息历史
- Agent 可在当前房间发言时引用其他房间的话题内容
- 支持通过配置开关控制是否启用跨房间感知
- 各房间的发言规则和并发调度机制保持不变

### V3.2: 以 Agent 为中心的事件驱动调度

重构调度模型，从房间视角改为 Agent 视角：

- 每个 Agent 持有独立的事件队列，被动接收来自各房间的新消息事件
- 调度器并发启动所有 Agent 的事件循环，不再控制发言顺序
- Agent 主动消费事件队列，自行决定何时在哪个房间发言
- 跨房间上下文感知能力保留（同 V3.1）
- 发言顺序由事件到达顺序自然决定，更贴近真实的异步通信模型

### V3.3: 自动化测试 (已完成)

引入单元测试与集成测试，为后续功能开发提供质量保障：

- 为核心模块（`agentService`、`roomService`、`schedulerService`、`funcToolService`）编写单元测试
- 使用 mock 隔离外部依赖（LLM API、消息总线），确保测试稳定可重复
- 编写集成测试，验证多 Agent 完整对话流程（消息路由、轮次调度、tool call 执行）
- 配置 CI 自动运行测试，覆盖率达到基准线
- 建立测试目录结构与命名规范，为后续版本测试提供模板

### V4: Web API 与实时可视化接入 (已完成)

在现有 service 层之上暴露 HTTP + WebSocket 接口，供可视化程序和外部系统实时接入：

- 提供 REST 接口，查询 Agent 列表、Room 列表及消息历史
- 提供 WebSocket 接口，订阅实时消息事件（Agent 发言、状态变更）
- 客户端无需轮询，服务端主动推送新消息
- 支持多客户端同时订阅，互不干扰
- 接口设计与 service 层解耦，不侵入核心调度逻辑

### V5: 终端可视化前端 (已完成)

基于 V4 的 HTTP/WebSocket 接口，开发终端 TUI 前端：

- 用 Textual 实现终端聊天室观察界面，风格类似桌面聊天软件
- 左侧房间列表 + 成员展示，键盘或鼠标切换房间
- 右侧实时滚动的对话气泡，不同 Agent 左右分列
- 通过 WebSocket 实时接收新消息，无需轮询
- 非当前房间有新消息时显示未读角标
- 纯观察模式（只读），与后端进程完全解耦

### V6: 交互式协作 (交互简化版)

引入人类“操作者”角色，并对房间进行类型化隔离以简化交互逻辑：

- **房间分类**：
    - **单聊 (Private)**：1 人类 + 1 Agent，采用严格的回合制交互，人类不发言 Agent 不动。
    - **群聊 (Group)**：多 Agent 自治模拟，人类仅作为观察者（同 V5）。
- **交互 TUI**：在单聊房间界面激活输入框，支持人类发送消息。
- **混合通信**：采用 HTTP POST 发送人类消息，WebSocket 实时订阅并展示回复。
- **逻辑简化**：避开多 Agent 并发抢答，优先打通“人类输入 -> Agent 定向回复”的核心链路。

### V7: 团队化组织与多租户隔离 (已完成)

引入 "Team" 概念作为顶层容器，支持更复杂的并发模拟场景：

- **顶层容器**：引入 Team 概念，将相关的 Agent 和 Room 进行逻辑分组管理。
- **资源标识符**：统一采用命名空间格式标识 Agent 和 Room，确保在多团队并发时的全局唯一性。
- **配置隔离**：不同 Team 拥有独立的成员名单、房间配置和运行参数，实现多租户级别的逻辑隔离。
- **多团队并发**：支持同时启动多个独立的团队模拟任务，各团队调度逻辑互不干扰。
- **扩展上下文模型**：同团队内的 Agent 跨房间共享对话感知，为复杂的跨场景协作奠定基础。

### V8: Agent 执行能力

集成 Claude Code Agent SDK，赋予 Agent 真实的执行能力，使其从"只会说话"升级为"能动手做事"：

- **执行能力**：Agent 可按配置读写文件、运行代码、搜索网络，并将执行结果以自然语言反馈到聊天室
- **混合协作**：普通对话 Agent 与具备执行能力的 Agent 可在同一聊天室共存协作

### V9: 数据持久化与重启恢复

在现有运行时能力之上补齐状态落盘与恢复机制，使服务在重启后可以恢复之前的状态，为后续手动继续运行提供基础：

- **状态持久化**：将 Team、Agent、Room、消息历史、事件队列和关键运行状态持久化到本地存储
- **重启恢复**：服务启动时自动加载最近一次快照或日志，恢复房间上下文、Agent 视图与调度状态
- **手动续跑**：恢复完成后默认不自动恢复聊天室调度，需由用户显式触发继续运行
- **消息连续性**：避免服务重启后丢失历史消息、未处理事件或房间成员关系，确保对话可连续进行
- **一致性保障**：为消息写入、事件消费和状态更新设计明确的落盘时机，降低异常退出导致的数据不一致风险
- **可观测与运维**：提供持久化开关、存储目录配置、状态检查与恢复日志，便于排查恢复过程中的问题

### V10: 组织树与部门管理

将 Team 内的成员关系从扁平列表升级为层级组织树，引入 Leader 与部门概念，支持对组织结构的动态编辑：

- **组织树模型**：Team 内成员以树形层级组织，每个节点可拥有任意数量的直接下属，根节点为团队最高负责人
- **Leader 与部门**：凡拥有至少一名直接下属的成员自动成为 Leader；Leader 与其所有直接下属共同构成一个部门
- **部门元数据**：每个部门可独立配置部门名称和部门职责描述文本，用于丰富 Agent 的上下文感知与 Prompt 构造
- **组织树编辑**：支持对现有组织树进行动态调整，包括新增下属关系、变更汇报线、提升或下移成员层级
- **休闲状态**：从组织树中移除某成员时，该成员不会被删除，而是进入"休闲（idle）"状态，保留其配置与历史上下文，可随时重新挂载到组织树中
- **上下文注入**：Agent 在生成回复时，自动感知所在部门的名称、职责和汇报链，为角色扮演与分工协作提供结构化背景

### V11: Agent 活动记录与运行态可观测性

为 Agent 增加独立的“活动记录（Activity Record）”对象，在运行过程中持续记录并广播关键行为，便于前端实时展示 Agent 正在做什么，并帮助定位运行期问题：

- **活动记录模型**：新增独立于消息历史的 `AgentActivityRecord` 领域对象，记录 `agent_id`、`team_id`、活动类型、状态、开始/结束时间、摘要文案、错误信息及扩展元数据；房间归属等可选上下文统一放入扩展元数据
- **关键活动覆盖**：至少覆盖大模型推理、工具调用、上下文压缩（compact）等高频运行阶段；context overflow 重试过程拆解为失败的推理、compact 和重试推理，而不是单独定义 `overflow_retry` 活动类型；调度等待不单独定义 `scheduler_wait` 活动类型，而是继续通过 Agent 的“激活 / 休眠”状态切换表达当前运行态，并在切换时追加对应的 `activate / idle` 活动记录
- **推理流式更新**：`llm_infer` 活动记录在模型流式返回时持续刷新当前已收到的 token 数，并向前端持续广播，便于实时展示推理进度
- **状态流转**：每条活动记录支持 `started / succeeded / failed / cancelled` 等状态，能够表达“开始执行”“执行成功”“执行失败”“被中断”这类生命周期
- **持久化存储**：活动记录写入独立存储，和消息历史解耦，支持按 Agent / Team / 时间窗口查询，并可按扩展元数据过滤房间等上下文，便于问题回放与运行审计
- **实时广播**：后端通过 WebSocket 主动广播活动事件，前端可实时订阅并展示 Agent 当前动作、耗时和失败原因，无需轮询推断
- **前端展示**：在 TUI / Web 前端中展示“Agent 正在推理”“正在调用工具”“正在压缩上下文”等运行态提示，降低用户对系统内部状态的不确定感
- **问题定位**：通过活动记录串联一次 turn 内的关键步骤，帮助快速判断问题发生在 LLM、tool、compact、调度还是持久化环节
- **消息历史解耦**：活动记录不等价于聊天消息，不直接进入对话上下文，避免为了调试和展示目的污染 Agent 的历史消息窗口

### V12: 大模型服务配置与动态切换

将 LLM 服务配置从静态文件提升为 Web Console 可视化管理能力，支持运行时动态切换：

- **服务列表管理**：在 Web Console 的"大模型服务管理"页面查看、新增、编辑、删除 LLM 服务配置，替换原有占位内容
- **默认服务切换**：运行时切换当前活跃的 LLM 服务，Agent 下一次推理自动使用新配置，无需重启后端
- **启用 / 禁用**：单独开关控制服务是否参与候选，禁用后不出现在可切换列表中
- **配置持久化**：Web Console 中的修改写回 `setting.json`，重启后配置不丢失
- **字段校验**：后端在保存前校验必填字段、URL 格式、类型枚举等
- **API Key 保护**：列表接口中 API Key 当前阶段明文返回，后续再加密

### V13: 快速初始化引导

为首次使用的用户提供自动化的配置向导，降低上手门槛：

- **启动检测**：后端启动时自动检测是否已配置大模型服务，若 `setting.json` 中无有效 LLM 配置则标记为"未初始化"状态
- **自动弹出**：用户访问 Web Console 时，若后端处于"未初始化"状态，自动弹出"快速初始化"引导弹窗，无需用户手动跳转
- **极简表单**：引导弹窗提供精简表单，仅要求用户填写必要信息（API 地址、API Key、模型名称），其他参数使用默认值或后续在管理页面调整
- **可用性测试**：配置完成后提供"测试连接"按钮，后端发起一次简单的推理请求验证服务可用性，实时反馈测试结果（成功 / 失败原因）
- **一键完成**：测试通过后点击"完成"按钮，自动保存配置到 `setting.json`，标记系统为"已初始化"状态，关闭弹窗进入正常工作界面
- **跳过引导**：支持"跳过"选项，用户可选择暂不配置，手动关闭弹窗进入主界面，后续通过"大模型服务管理"页面完成配置
- **系统状态接口**：提供系统状态查询接口（`/system/status.json`），返回包含初始化状态在内的系统运行状态，前端据此判断是否需要触发引导流程
- **TUI 兼容**：TUI 前端通过系统状态接口查询初始化状态，未初始化时提示用户手动编辑 `setting.json` 或通过 Web Console 完成配置

### V14: Agent 团队感知与协作能力

为 Agent 提供一组内置工具函数，使其能够主动感知所在团队的组织结构、房间状态和同伴运行状况，并在必要时唤醒失败的同伴，实现 Agent 层面的自主协作：

- **组织与房间查询**：Agent 可通过工具函数查询当前所属 Team 的基本信息、组织树结构、房间列表及房间内成员，在对话中自然地引用团队上下文
- **同伴状态查询**：Agent 可查询同 Team 内其他 Agent 的当前运行状态（idle / running / failed 等），了解同伴是否在线、是否正常工作
- **失败唤醒（wake_up）**：当 Agent 发现同伴处于 failed 状态时，可调用 `wake_up` 工具函数请求后端重新激活该 Agent，触发其重新进入调度循环
- **安全约束**：`wake_up` 仅允许唤醒同 Team 内的 Agent，且目标 Agent 必须处于 failed 状态；对非法调用返回明确错误信息，防止滥用
- **工具注册**：新增工具通过现有 `funcToolService` 机制注册，使用 `_context` 注入调用者身份，复用已有的权限与上下文隔离能力
- **Prompt 增强**：在 Agent 的系统提示中注入团队感知工具的使用说明，引导 Agent 在合适场景下主动查询和协作，而非盲目等待调度

### V15: 房间工作中状态展示

优化房间中的运行态反馈，让用户在 Agent 处理房间任务时能够明确看到“当前谁在工作”，降低长时间无消息时的理解成本：

- **工作中可见**：当 Agent 正在处理房间任务时，界面能明确显示当前工作中的 Agent，而不是完全静默
- **私聊与群聊都可见**：私聊房间和群聊房间都提供一致的“当前处理中”反馈，但不改现有消息写入逻辑
- **失败状态可识别**：若当前处理中的 Agent 失败，界面上的工作中提示能够及时结束或切换为失败状态，避免误导用户
- **前端体验统一**：Web / TUI 前端都能看到一致的工作状态提示，不依赖额外手动刷新
- **范围可控**：该版本先解决展示层可见性问题，不引入消息排队、消息状态升级等更重的数据模型改造

### V17: 人工干预与 Agent 停止能力

为人类操作者提供实时干预手段，当 Agent 执行过程中出现异常或不符合预期的行为时，可随时强制中断，避免无效消耗或错误扩散：

- **停止单个 Agent**：Web/TUI 界面提供"停止"操作入口，人类可在任意时刻中断指定 Agent 的当前执行任务（推理、工具调用均可中断）
- **后端接口**：新增 HTTP 接口，接收停止请求后向目标 Agent 发送取消信号，终止当前正在执行的 turn
- **状态流转**：被停止的 Agent 当前 turn 以 `cancelled` 状态结束，Agent 本身回到 `IDLE` 状态，可继续正常参与后续调度
- **私聊消息收尾**：私聊房间中若有 `TYPING` 占位消息，停止时按取消流程处理，避免消息永远停留在输入中状态
- **上下文感知**：停止后向 Agent 对话历史写入中断说明，Agent 下次被调度时能感知到上一轮未完成，不会基于残缺上下文产生混乱
- **活动记录**：利用 V11 的活动记录机制记录"被人工停止"事件，保留操作痕迹
- **范围约束**：本版本只支持停止单个 Agent 的当前任务，不提供批量停止或停止整个 Team 的能力

### V16: 私聊消息排队与占位提交（暂缓）

在 V15 的展示反馈基础上，进一步升级私聊房间的消息模型，使 Agent 的输入中状态直接体现在消息流中，并在完成后原地变为最终消息：

- **私聊占位消息**：当 Agent 处理私聊房间发言任务时，消息流中出现明确的输入中占位
- **原地变为结果**：Agent 完成当前轮发言后，占位消息在原位置变为最终消息，不再额外新增一条重复消息
- **消息顺序稳定**：私聊房间中后续消息与占位消息保持自然时序，不出现完成后跳位或乱序
- **异常流程闭环**：当 Agent 失败、取消或未产出有效回复时，占位消息能被正确收尾，避免长期停留在输入中状态
- **支持重启恢复**：相关状态进入持久化模型，重启后仍能恢复私聊消息的可见性和推进状态

### V18: 国际化（i18n）支持

为前端界面和内置预置内容提供多语言支持，首批覆盖中文和英文，架构上允许后续仅通过新增资源文件即可扩展其他语言：

- **前端界面多语言**：Web 前端所有用户可见的界面文案（菜单项、按钮标签、提示信息、状态文本、对话框标题等）从硬编码中文抽取为翻译 key，通过 i18n 资源文件加载
- **语言切换**：Web Console 顶栏和 macOS 系统托盘菜单均提供语言切换入口，用户可在中文 / 英文之间即时切换，语言偏好存储在后端配置中，多端共享
- **资源文件组织**：翻译资源按语言代码组织（如 `zh-CN.json`、`en.json`），新增语言时只需添加对应资源文件，无需修改框架代码或组件逻辑
- **Preset 多语言**：内置预置内容采用单文件内嵌多语言方案（`display_name` 多语言字段），导入时将多语言数据存入 DB 的 `i18n` 字段，语言切换时直接读取对应版本
- **Preset 去重**：Team 配置携带 UUID，防止切换语言后重复导入
- **TUI 兼容**：TUI 前端的固定文案（面板标题、状态标签、提示文字）同步支持多语言，复用相同的翻译 key 体系
- **扩展性保障**：架构设计确保新增语言（如日文、韩文等）只需添加资源文件和预置翻译，不涉及代码结构变更或数据库迁移

### V19: HTTP API 鉴权

为后端 HTTP API 与 WebSocket 接口提供可选的鉴权能力，防止未授权访问：

- **可选启用**：鉴权功能通过 `setting.json` 中的 `auth.enabled` 字段控制，默认关闭，向后兼容
- **Token 认证**：用户输入访问 token（配置文件中预设），前端存储 token 后在每次请求中携带
- **全接口覆盖**：启用鉴权后，除系统状态接口外的所有 HTTP API 需携带 token，WebSocket 通过认证消息发送 token
- **前端自动认证**：前端自动检测鉴权状态，输入 token 后自动存储并在请求中携带，WebSocket 连接后发送认证消息
- **统一错误响应**：鉴权失败统一返回 HTTP 401 状态码，包含标准化的错误码和中文描述

### V20: 运行中上下文插入能力

- 私聊房间中 Agent 工作期间发送的普通消息，在 Agent 本轮结束后才出现（出现在 Agent 回复之后）
- 私聊房间中发送消息时，可选择"即时插入"，在 Agent 当前步骤完成后立即注入上下文
- 工具执行期间即时消息不中断当前动作，等当前工具完成后才注入
- 仅支持私聊房间；当前先支持 host-managed 工作流

### V21: Agent 监督指令与控制房间

在 V20 的基础上，为人类操作者提供真正可用的“运行中追加指令”体验，使其能够一边观察 Agent 的执行过程，一边及时纠偏：

- **活动窗口直接下指令**：操作者可在 Agent 的状态 / 活动窗口中直接输入补充要求、纠偏建议或临时约束，无需切换到别的界面
- **首次使用自动建通道**：当操作者第一次给某个 Agent 发送追加指令时，系统自动建立对应的私聊控制房间，避免默认界面一开始就堆满大量单聊
- **指令尽快生效**：追加指令发出后，会在当前工作过程里尽快影响 Agent 的后续行为，而不是只能等整轮完全结束后才起作用
- **正式沟通规则不变**：操作者虽然能通过活动记录观察 Agent 的执行轨迹，但 Agent 若要正式回复操作者或团队，仍需通过房间消息明确表达，避免“看得到执行过程”与“正式沟通”混淆
- **全过程可追溯**：追加指令及其后续响应都会保留在对应的私聊控制房间中，便于回看一次干预是如何发生、又带来了什么结果

### V22: Agent Task 系统

为 Agent 团队引入轻量级的结构化任务管理能力，使 Agent 间协作从"消息流中约定"升级为"可追踪、可验收的正式任务"：

- **结构化任务创建**：Agent 可通过工具创建任务，显式指定执行人、验收人、优先级和依赖关系
- **完整状态机**：任务经历 TODO / PENDING / IN_PROGRESS / ON_HOLD / REVIEWING / DONE / CANCELLED 状态，变更操作受权限约束
- **依赖自动协调**：前置任务未完成时新任务自动进入 PENDING，系统拒绝跳过依赖直接开始
- **验收流程**：有 manager 的任务须经 assignee 提交 → manager 审核，支持打回返工
- **全团队透明**：团队内所有任务对所有 Agent 可见，按 team_id 隔离；本版本仅提供 Agent 工具层，不含前端界面
